## What is D.U.M.M.Y. OS?

D.U.M.M.Y. OS is an **AI Operating System** вҖ” not an app, not a plugin, not a chatbot.

It's a layer that installs inside any AI tool (Claude Code, Cursor, Gemini CLI) and transforms it into a fully orchestrated, self-correcting, memory-persistent system.

You speak your intent. D.U.M.M.Y. figures out the rest.

```
"I want a task manager app with Google login and a database"
         вҶ“
D.U.M.M.Y. OS
  вҶ’ ConnectPro sets up Supabase automatically
  вҶ’ app-factory builds the full app
  вҶ’ preview-bridge opens it running in your browser
  вҶ’ surge-core fixes any error before you notice it
         вҶ“
Working app. One prompt.
```

## Install

```bash
npx dummy-os install
```

Detects Claude Code, Cursor, or Windsurf and installs all 8 processes automatically.

Boot the OS in any session:

```
hi dummy
```

The OS loads your project memory and waits for your next intent.

## What's new in v2.2

### Session persistence вҖ” hook system
The OS now uses a `UserPromptSubmit` hook that injects a kernel signal into **every message**. The OS can no longer drift or fall back to default AI behavior mid-session. Once active, it stays active.

### ConnectPro вҖ” 3-file architecture
ConnectPro is now split into:
- `SKILL.md` вҖ” pure policy and decision tree
- `CONNECTORS.md` вҖ” per-service playbooks (Supabase, Stripe, Google, Vercel, GitHub, n8n)
- `BROWSER_AUTO.md` вҖ” Playwright scripts per service + Email Loop

### Email Loop
When a service sends a verification email (Stripe, Supabase, Google, GitHub), ConnectPro automatically reads it via Gmail MCP, extracts the code or link, and completes the verification вҖ” without asking you to open your inbox.

### AUTONOMY_POLICY
ConnectPro now has 3 explicit autonomy levels:
- **Silent**: read data, detect services, extract keys from logged-in dashboards
- **Confirm first**: create projects that generate cost, issue permanent tokens
- **Never**: delete existing resources, revoke credentials

### Workflow Engine (Tipo G)
The orchestrator now routes automation requests:
- Internal automations вҶ’ `scheduled-tasks` MCP (cron jobs)
- External workflows (between services) вҶ’ `n8n` MCP
- "Every time X, do Y" вҶ’ handled without creating new skills

### Formal specs
- `surge-core/SURGE_CORE_SPEC.md` вҖ” explicit autonomy limits, code auditor, decision tree
- `dummy-memory/SPEC.md` вҖ” formal read/write contract, TTL, conflict policy, secret protection

### Security
- `.dummy/memory/` is now in `.gitignore` вҖ” credentials referenced in `env.md` can never accidentally reach a git repository.

### Token optimization
Skill files cut by ~50%: orchestrator 321вҶ’161 lines, dummy-memory 272вҶ’139 lines. Fewer tokens consumed per session without losing any functionality.

## Memory System

D.U.M.M.Y. OS remembers between sessions:

```
.dummy/memory/
  projects/{name}/
    state.md      вҖ” what was built, which routes/components exist
    env.md        вҖ” which credentials were resolved (never the values)
    decisions.md  вҖ” architectural decisions and autonomy actions
    errors.md     вҖ” errors fixed (never repeated)
  user/
    preferences.md вҖ” language, frameworks, coding style
  global/
    errors.md     вҖ” reusable error patterns across projects
```

The kernel loads relevant memory at the start of every session automatically.
`.dummy/memory/` is protected by `.gitignore` вҖ” credentials never reach git.

```bash
dummy init         # initialize memory for current project
dummy status       # show installed skills + project memory
dummy memory load  # display what the OS knows about your project
dummy memory clear # reset project memory
```
